# WriteFlow 构建与部署说明

本文档用于说明当前仓库的构建、版本管理与发布流程。

## 1. 仓库结构

- `packages/*`：可发布的库包（核心包 + 扩展包 + react 包）
- `examples`：示例站点，仅用于本地调试，不参与库发布
- `turbo.json`：Turbo 任务编排
- `tsup.config.ts`：库包统一打包配置
- `.changeset/*`：版本与发布元数据
- `.github/workflows/*`：CI 与自动发布流程

## 2. 构建策略

### 2.0 工具职责说明

#### tsup 的作用

`tsup` 是单个包的打包器，负责把每个库包从 `src` 编译成可发布产物：

- 将 TypeScript 编译为 `esm` 和 `cjs`
- 生成类型声明 `d.ts`
- 统一输出到 `dist`
- 通过 `esbuild-sass-plugin` 处理 `scss` 导入

在本仓库中，`tsup` 由每个包的 `build` 脚本触发，且共享根配置 `tsup.config.ts`。

#### turbo 的作用

`turbo` 是 monorepo 任务编排器，不负责具体编译，而是负责“按依赖关系调度任务”：

- 识别包之间依赖顺序（先构建被依赖包，再构建依赖方）
- 并行执行可并行的包构建任务
- 基于 `outputs` 做任务缓存，减少重复构建时间
- 通过 `--filter=./packages/*` 精准控制构建范围

在本仓库中，`turbo` 调度的是各包的 `build` 脚本，而真正产出文件的是 `tsup`。

### 2.1 构建范围

只构建 `packages/*`，不构建 `examples`。

根命令：

```bash
pnpm build:packages
```

对应脚本：

- `package.json` -> `turbo run build --filter=./packages/*`
- `turbo.json` -> `build` 任务依赖 `^build`，输出 `dist/**`

### 2.2 包产物格式

每个包统一通过 `tsup` 构建，产物在各包 `dist` 目录下：

- `dist/index.js`（ESM）
- `dist/index.cjs`（CJS）
- `dist/index.d.ts`（类型声明）

构建配置统一由根目录 `tsup.config.ts` 提供：

- 入口：`src/index.ts`
- 格式：`esm,cjs`
- 类型：`dts: true`
- 样式：通过 `esbuild-sass-plugin` 支持 `scss`，并通过 `injectStyle: true` 将样式自动注入入口 JS，消费者无需再额外手动导入库样式文件
- TypeScript 配置：显式使用 `tsconfig.build.json`（避免继承业务 tsconfig 中的 `incremental/noEmit/next plugin` 影响 dts 构建）

```

```

## 4. 版本管理（Changesets）

本仓库使用 Changesets 管理多包版本。

配置文件：`.changeset/config.json`

关键配置：

- `baseBranch: "main"`
- `access: "public"`
- `ignore: ["@kunlunxu/wf", "@kunlunxu/wf-examples"]`
- `updateInternalDependencies: "patch"`

### 4.1 提交功能时

在功能 PR 中创建 changeset：

```bash
pnpm changeset
```

根据改动选择：

- `patch`：修复或兼容性小改动
- `minor`：向后兼容的新功能
- `major`：不兼容变更

会生成 `.changeset/*.md` 文件，需一起提交。

## 5. 发布流程

### 5.1 自动发布（推荐）

触发条件：代码进入 `main` 分支，且仓库中存在未消费的 `.changeset/*.md`。

当前 `release.yml` 不再创建 Release PR，而是在 `main` 上直接发布：

1. 开发时在本地运行 `pnpm changeset`
2. 生成 `.changeset/*.md` 文件，并和代码一起提交
3. 将代码推送到 `main`，或通过普通功能 PR 合并到 `main`
4. GitHub Actions 检测到未消费的 changeset 后，自动执行 `pnpm version-packages`
5. workflow 自动更新包版本、生成 changelog，并将这些版本变更直接提交回 `main`
6. 随后执行 `pnpm release`，发布到 npm
7. 最后推送本次发布产生的 git tags

说明：

- `.changeset/*.md` 可以在本地生成，但只有进入 `main` 后才会触发自动发布
- 自动生成的版本提交使用 `[skip ci]`，避免版本提交再次触发一轮 release
- 如果当前推送不包含未消费的 `.changeset/*.md`，release workflow 会直接跳过

### 5.2 必要 Secrets

在 GitHub 仓库中配置：

- `NPM_TOKEN`：npm 发布令牌
- `GITHUB_TOKEN`：Actions 自动提供；当前 workflow 使用它把版本提交和 tags 推回仓库

### 5.3 手动发布（应急）

```bash
pnpm version-packages
pnpm build:packages
pnpm release
```

说明：手动发布前请确保当前分支与 `main` 同步，且 CI 已通过。

## 6. CI 流程

文件：`.github/workflows/ci.yml`

触发：`pull_request` 和 `push main`

当前校验项：

1. 安装依赖
2. 执行 `pnpm build:packages`

备注：当前工作流使用 `pnpm install --no-frozen-lockfile`。  
建议在依赖稳定后切换为 `--frozen-lockfile`，提升可复现性。

## 7. 发布排障

### 7.1 没有发布任何包

常见原因：没有提交 `.changeset/*.md`。  
处理：执行 `pnpm changeset` 并提交变更。

### 7.2 npm 发布权限失败（403）

检查：

- `NPM_TOKEN` 是否过期
- 包名是否归属当前 npm 组织/账号
- `publishConfig.access` 是否为 `public`

### 7.3 回滚策略

npm 不支持覆盖已发布同版本。建议：

1. 使用 `npm deprecate` 标记坏版本
2. 通过 changeset 发布新补丁版本修复
